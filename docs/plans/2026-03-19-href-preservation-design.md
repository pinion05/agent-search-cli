# Href Preservation in Agent-Browser-Like Pruning Design

## Goal

Preserve real link destinations in the reduced HTML generated by `agent-browser-like-pruning` without regressing the accepted text/content/interaction signals.

## Problem

The current reducer keeps structure labels, but `renderReducedDocumentHtml()` rewrites every structure item as:

```html
<a href="#">…</a>
```

That loses the original destination and can mislead downstream review because the reduced HTML visually suggests a link while discarding the actual target.

## Constraints

- keep the change as small as possible
- do not add new dependencies
- preserve current accepted signal behavior for identity, facts, content, and interactions
- avoid fake `#` links for items that were never links
- make reduced HTML links usable when opened from `file://`

## Options

### Option A — keep `string[]` items and add a side map

- keep `ReducedDocumentSection.items: string[]`
- add a separate label-to-href map somewhere else in `ReducedDocument`

**Pros**
- smallest local type change in existing tests

**Cons**
- duplicates label matching state
- makes renderer indirect and fragile
- ambiguous when the same label appears more than once

### Option B — replace structure strings with item objects

- introduce a structure item shape such as:

```ts
type ReducedDocumentStructureItem = {
  text: string;
  href?: string;
};
```

- change `ReducedDocumentSection.items` from `string[]` to `ReducedDocumentStructureItem[]`
- render `<a href="...">` only when `href` exists
- render plain text for structure items that do not map to a real anchor

**Pros**
- smallest coherent data-flow change
- renderer gets everything it needs directly
- supports mixed sections where some items are real links and others are plain structure labels
- avoids fake anchors

**Cons**
- requires test updates where assertions currently expect raw strings

### Option C — preserve hrefs everywhere (`structure`, `content`, `interactions`)

- broaden the reduced schema so any preserved line can carry a destination

**Pros**
- richest future-proof model

**Cons**
- larger surface area
- more matching ambiguity
- higher regression risk for this phase

## Recommendation

Choose **Option B**.

It is the smallest schema/data-flow that actually preserves destinations in reduced HTML while keeping the rest of the reducer behavior stable.

## Chosen Shape

```ts
type ReducedDocumentStructureItem = {
  text: string;
  href?: string;
};

type ReducedDocumentSection = {
  heading: string;
  items: ReducedDocumentStructureItem[];
};
```

## Data Flow

1. Parse `document.bodyHtml` with existing `linkedom` (`parseHTML`), not a new library.
2. Collect anchor candidates in DOM order:
   - normalized visible text
   - raw `href`
   - resolved absolute `href` using `new URL(href, document.finalUrl || document.url).toString()`
3. Let the existing mode heuristics continue deciding **which structure labels to keep**.
4. After a label is selected for a section item, try to attach an `href` by matching the selected text against collected anchors:
   - exact normalized text match first
   - case-insensitive normalized match second
   - consume first unused matching anchor to avoid duplicate-label collisions
5. For labels that came from buttons or non-anchor snapshot text (for example map selected-panel text), keep `href` undefined.
6. In `renderReducedDocumentHtml()`:
   - render `<a href="resolved-url">text</a>` when `href` exists
   - otherwise render plain text in the list item instead of `<a href="#">`

## Scope Boundary

This phase should only preserve real destinations for the **structure** layer.

Do **not** expand href preservation into `facts`, `content`, or `interactions` in this pass unless a failing acceptance case proves that structure-only preservation is insufficient.

## Why Absolute URLs

Reduced review files are opened from `file://.../reduced.html`.

If we keep relative paths like `/docs`, the link destination becomes the local filesystem root instead of the original site route. Resolving against `finalUrl` preserves the real navigation target in the reduced artifact.

## Expected File Ownership / Handoff

### Worker-2 implementation lane

- `src/types.ts`
- `src/agent-browser-like-pruning.ts`
- `src/agent-browser-like-pruning.test.ts`

### Worker-3 verification lane

- run typecheck/tests
- refresh sample outputs
- if needed, update review/sample fixtures so `bodyHtml` actually contains href-bearing anchors for preservation checks

## Suggested Tests

1. a docs/package/place-detail fixture where `bodyHtml` includes real anchor hrefs and reduced structure items keep the resolved destination
2. a render test proving:
   - linked items render real `href`
   - non-linked structure items render plain text, not `#`
3. a duplicate-label test proving first-unused DOM-order matching stays stable
4. a relative-link test proving `/docs` becomes an absolute URL based on `finalUrl`

## Non-Goals

- no redesign of the overall reduced document pipeline
- no new interaction-link model
- no fuzzy label matching
- no doc-wide link graph extraction

## Team Decision

Default to Option B unless the leader or worker-2 objects with a concrete counterexample.
