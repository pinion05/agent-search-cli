# brAIve 통합 설계

**날짜:** 2026-03-14

> 이 문서는 `brAIve` v1 설계의 현재 단일 기준 문서(source of truth)다.
>
> 다음의 이전 초안 방향성을 대체한다:
>
> - `docs/plans/2026-03-13-agent-search-cli-design.md`
> - `docs/plans/2026-03-13-agent-search-cli-implementation.md`
> - GitHub 이슈 `#7` 의 이전 코멘트들

## 1. 제품 정의

`brAIve` 는 공개 웹페이지를 llm 의 이해를 최우선으로 압축된 `.toon` 아티팩트로 변환하는 로컬 CLI다.

v1에서 이 제품은 기본적으로 검색 요약기가 아니다. 검색은 여러 입력 경로 중 하나일 뿐이다. 핵심 가치는 다음과 같다.

- 사람이 보는 방식대로 페이지를 렌더링한다
- 노이즈가 많은 DOM을 작고 읽기 쉬운 중간 구조로 축약한다
- 그 구조를 TOON으로 직렬화한다
- 결과를 재사용할 수 있도록 로컬에 캐시한다

v1에서 주요 아티팩트는 **페이지 단위 `.toon` 파일**이다.

## 2. 핵심 원칙

`brAIve` 는 **무료 Brave Search API 티어를 우선 기준**으로 설계된다.

기본 경로는 다음 환경에서도 충분히 유용해야 한다.

- 무료 Brave Search API 키
- 로컬 머신 실행

유료 Brave AI 엔드포인트는 지원하지 않는다.

## 3. 비목표

v1의 목표에는 다음이 포함되지 않는다.

- Brave AI 엔드포인트 통합을 기본 경로로 제공하는 것 (영구 비목표)
- 최종 다중 페이지 합성 (추후계획)
- 최종 LLM 요약 생성을 주요 제품 출력으로 삼는 것
- 로그인 보호 사이트 크롤링
- 안티봇 우회 시스템
- 무한 스크롤 중심 앱 지원
- PDF/OCR 수집

## 4. 지원 입력

CLI는 첫날부터 세 가지 입력 모드를 지원한다.

- `braive url <URL>`
- `braive urls <URL...>`
- `braive query "<search query>"`

### 옵션

`url` 과 `urls`:

- `--refresh`

`query`:

- `--count`
- `--refresh`

### query 기본값

- 기본 검색 결과 개수: 5개
- `--count` 에 하드 상한 100개
- `urls` 입력 길이에 하드 상한 100개 동일

## 5. Query 동작

`query` 모드는 Brave Web Search만 사용한다.

규칙:

- Brave 검색 결과는 반환된 순서를 그대로 사용한다
- 도메인 중복 제거는 하지 않는다
- 최소한의 필터링만 적용한다:
  - 빈 URL 제거
  - 명백히 잘못된 URL 제거
  - `javascript:` 형태의 사용 불가능한 링크 제거
  - 명백한 리디렉트 전용 중간 링크 제거
- Brave 검색 결과는 **캐시하지 않는다** (toon 을 캐시하는것이 더 효율적이기 때문입니다)
- Brave 검색이 실패하면 전체 실행이 실패한다
- Brave 검색이 사용 가능한 결과를 하나도 반환하지 않으면 시스템 stderr 는 아니고 out 으로 각각 명시적 에러표시한다.

## 6. 실행 모델

### 순서

- `query` 모드에서 urls 레이어로 전달될때 url 의 순서는 Brave api 결과 순서를 유지한다 (seo 의 추천우선순위를 지키기위함)
- `url` 과 `urls` 는 입력 순서를 유지한다
- 중복된 canonical URL은 한 번만 처리한다
- 최종 출력은 원래 입력 순서대로 렌더링한다

### Canonical URL

캐시 키와 중복 제거에 사용되는 canonical URL은 입력 URL에서 파생한다. `final_url`(리다이렉트 결과)이 아니다.

규칙:

1. scheme을 소문자로 변환한다
2. host를 소문자로 변환한다
3. fragment를 제거한다
4. 그 외 경로, query parameter, trailing slash는 변경하지 않는다

### 동시성

- 최상위 페이지 처리는 병렬로 실행된다
- 기본 동시성: `3`

### 브라우저 재사용

- 단일 실행 동안 브라우저 인스턴스 하나를 재사용한다
- 각 최상위 URL은 자신만의 페이지/탭을 가진다

### 프레임 처리

* same-origin iframe은 부모 프레임 컨텍스트에서 순회한다
* cross-origin iframe은 그 `src` 를 통해 다시 방문한다
* 재귀적 iframe 재방문 깊이의 기본값은 `2` 다
* iframe 콘텐츠는 별도 출력 문서로 내보내지 않고 부모 문서 위치에 병합한다

## 7. 렌더링 정책

페이지별 기본 렌더링 순서는 다음과 같다.

1. `domcontentloaded` 까지 대기
2. `1000ms` 대기
3. 자동 스크롤 1회 수행
4. `500ms` 대기
5. DOM 캡처

### 타임아웃

- 페이지별 타임아웃: `15s`

### 타임아웃 동작

- 타임아웃이 발생해도 페이지를 자동 폐기하지 않는다
- 사용 가능한 DOM이 존재하면 `brAIve` 는 여전히 출력을 생성한다
- 미완성 영역은 실패 노드로 표현한다

### HTTP 오류 동작

- 2xx가 아닌 응답이라고 해서 페이지를 자동 폐기하지 않는다
- 응답 본문에 읽을 수 있는 콘텐츠가 있다면 여전히 출력으로 축약될 수 있다

### robots

- v1은 `robots.txt` 를 명시적으로 해석하지 않는다

## 8. 보안 경계

이 경계들은 구현 완료로 간주되기 전에 반드시 정의되어야 한다.

### URL 스킴

허용되는 입력 스킴:

- `http://`
- `https://`
- `file://`

알 수 없거나 사용할 수 없는 스킴은 일반적인 실패 처리로 종료된다.

### 명시적 위험 영역

`file://` 이 허용되고 cross-origin iframe의 `src` 도 자동으로 다시 방문할 수 있기 때문에, 별도 제한이 없으면 크롤러가 안전하지 않은 신뢰 구역으로 흘러들어갈 수 있다.

따라서 구현에서는 다음을 정의하고 강제해야 한다.

- 허용 스킴 화이트리스트
- 네트워크 재방문 시 loopback 및 메타데이터 주소 제한
- 내부 네트워크 차단 정책
- `file://` 이 재귀적으로 네트워크 콘텐츠를 참조할 수 있는지 여부

### 브라우저 격리

구현은 실행 단위 격리 경계를 정의해야 한다.

- 실행마다 새로운 격리 브라우저 컨텍스트를 사용하거나
- 그와 동등하거나 더 엄격한 방식

서로 무관한 대상 간에 cookie/localStorage/service-worker가 새어 나가는 문제를 명시하지 않은 채 둘 수 없다.

### 로컬 저장소 보호

TOON 캐시 디렉터리는 제한적인 권한으로 생성되어야 한다.

최소 기대 수준:

- `~/.brAIve/toon-cache/` 는 사용자 전용이어야 한다

## 9. DOM 축약 철학

목표는 **DOM 충실도**가 아니다. 목표는 **사람이 인지하는 정보 충실도**다.

즉 다음을 의미한다.

- 사람이 읽고 사용할 수 있는 것을 보존한다
- 구현 전용 구조는 버린다
- 순서와 중첩은 의미 전달에 도움이 되는 범위에서만 유지한다

### 제거

- `id`
- `class`
- inline style
- 프레임워크/내부 속성
- 레이아웃 전용 래퍼
- 비어 있는 래퍼
- 공백만 있는 노드

### 보존

- 읽을 수 있는 텍스트
- `href`
- `src`
- `alt`
- 축약된 의미적 중첩을 반영하는 구조 깊이

## 10. 축약 파이프라인

렌더링된 DOM에서 최종 캐시 아티팩트까지의 파이프라인은 다음과 같다.

1. 렌더링된 DOM
2. 의미 없는 래퍼 제거
3. 축약 트리 구성
4. 축약 트리에 대해 preorder 순회
5. 균일한 일반화 노드 배열 생성
6. TOON으로 인코딩

일반화 계층은 의도적으로 세밀하게 유지한다.

규칙:

- 인접한 콘텐츠를 과도하게 병합하지 않는다
- 문장 안에 링크가 포함되면 `text / link / text` 로 분리한다
- 표와 코드 블록은 특별한 노드 종류가 아니라 텍스트로 표현한다

## 11. 일반화 JSON 스키마

TOON 변환 전의 정식 JSON 형태는 다음과 같다.

```json
{
  "url": "https://example.com",
  "final_url": "https://example.com/docs",
  "title": "Example Docs",
  "fetched_at": "2026-03-14T10:00:00Z",
  "nodes": [
    {
      "idx": 0,
      "deps": 0,
      "text": "Example Docs",
      "href": "",
      "src": "",
      "alt": "",
      "status": ""
    }
  ]
}
```

### 최상위 필드 순서

- `url`
- `final_url`
- `title`
- `fetched_at`
- `nodes`

### 노드 필드 순서

- `idx`
- `deps`
- `text`
- `href`
- `src`
- `alt`
- `status`

### 노드 의미

`idx`

- preorder 순회 순서

`deps`

- 축약 트리 깊이
- 원본 DOM 깊이가 아님

`text`

- `innerText` 에서 파생
- HTML 엔티티 디코드
- 앞뒤 공백 제거
- 내부 텍스트 흐름은 가능한 한 유지

`href`

- 원본 소스 값
- 절대 URL로 변환하지 않음

`src`

- 원본 소스 값
- 절대 URL로 변환하지 않음

`alt`

- 원본 alt 텍스트가 있으면 그 값, 없으면 `""`

`status`

- 일반 노드는 `""`
- 실패 노드는 `"fail"`

### 노드 생성 규칙

다음 둘 중 하나를 만족하면 노드를 생성한다.

- `trim(text) != ""`
- 또는 `href`, `src`, `alt` 중 하나라도 비어 있지 않음

앞뒤 공백 제거 후 살아남는 한 글자 텍스트 노드도 허용한다.

## 12. 페이지 메타데이터 규칙

`url`

- 원래 입력값 또는 해석된 소스 URL 식별자

`final_url`

- 최종 이동된 URL
- 사용할 수 없으면 빈 문자열

`title`

- `<title>` 태그만 사용
- 대체 휴리스틱 없음
- 없으면 빈 문자열

`fetched_at`

- 항상 존재
- ISO 8601 UTC

## 13. 실패 표현

실패하더라도 `.toon` 파일은 생성된다.

최소 실패 형태:

```json
{
  "url": "bad-url",
  "final_url": "",
  "title": "",
  "fetched_at": "2026-03-14T10:00:00Z",
  "nodes": [
    {
      "idx": 0,
      "deps": 0,
      "text": "invalid url",
      "href": "",
      "src": "",
      "alt": "",
      "status": "fail"
    }
  ]
}
```

규칙:

- 실패 결과도 실제 `.toon` 파일을 기록해야 한다
- stdout에 출력하는 경로는 항상 실제로 존재해야 한다
- 실패 사유는 첫 번째 노드의 `text` 에 넣는다

## 14. TOON 변환

TOON은 개념적 원본 모델이 아니라 최종 전송/저장 형식으로 취급한다.

역할은 다음과 같다.

- 평탄한 일반화 JSON을 압축한다
- 균일한 객체 배열 압축을 활용한다
- 후속 소비자의 토큰 사용량을 줄인다

현재 일반화 형식은 TOON에 잘 맞도록 의도적으로 설계되어 있다.

- 평탄한 최상위 구조
- 하나의 균일한 `nodes` 배열
- 모든 노드에 동일한 필드 집합

## 15. 캐시 정책

단일 저장 루트:

- `~/.brAIve/toon-cache/`

규칙:

- TOON 파일만 영속 캐시 아티팩트로 저장한다
- 별도 query 캐시는 없다
- 별도 출력 디렉터리는 없다
- TTL: `1 day`
- 키 기준: 입력 URL에서 파생한 canonical URL
- 파일명 형식: URL 기반의 파일 안전 slug + 짧은 hash suffix + `.toon`

예시:

- `https___example.com_docs__a1b2c3.toon`

### Refresh

- 기본 동작: 유효한 TOON 캐시를 재사용
- `--refresh`: 해당 파일을 강제로 다시 기록

### 안전한 쓰기

구현이 시작되기 전, 이 설계는 다음을 요구하는 것으로 간주해야 한다.

- 임시 파일에 먼저 기록한 뒤 atomic rename 수행
- 읽기 시 잘못되었거나 손상된 TOON 감지
- 파싱 실패 시 캐시 무효화

## 16. CLI 출력 계약

치명적이지 않은 모든 실행 결과는 점진적으로 스트리밍하지 않고 `stdout` 으로 출력한다.

규칙:

- 모든 결과 라인을 메모리에 수집
- 입력 순서대로 정렬
- 마지막에 한 번만 출력
- `stderr` 는 진짜 프로세스 수준 치명적 오류에만 사용

### Query 모드 헤더

`query` 모드만 헤더를 출력한다.

```text
query: OpenAI Codex
```

### 결과 라인 형식

- `saved: URL -> PATH`
- `cached: URL -> PATH`
- `refreshed: URL -> PATH`
- `failed: URL -> PATH (reason)`

규칙:

- 경로는 `~` 축약 형식 사용
- 입력 항목당 한 줄
- 실패 항목도 `.toon` 경로를 포함해야 한다

## 17. 성공과 실패 의미

### `query` 모드가 실패하는 경우

- Brave 검색 실패
- Brave 검색이 사용 가능한 결과를 하나도 반환하지 않음
- 후속 페이지 결과가 전부 실패함

### 혼합 결과

- 페이지 하나라도 성공하면 전체 명령은 성공으로 간주한다

### 잘못된 URL

- `url` 모드: 잘못된 입력이면 전체 실행 실패
- `urls` 모드: 잘못된 항목은 개별 실패 처리하고 나머지는 계속 진행

## 18. 구현 전 테스트 요구사항

이 통합 설계에 맞춰 테스트를 다시 계획하기 전에는 구현을 시작하면 안 된다.

향후 테스트 전략은 최소한 다음을 다뤄야 한다.

- 모드별 CLI 계약 (`url`, `urls`, `query`)
- 최종 stdout 렌더링 계약
- 캐시 적중 / refresh / 실패 파일 기록
- 크래시 안전 쓰기 동작
- Playwright 렌더링 플로우
- same-origin iframe 임베딩
- cross-origin iframe 재방문 동작
- `file://` 처리 동작
- 축약 트리 생성
- 일반화 스키마 검증
- TOON 출력 생성
